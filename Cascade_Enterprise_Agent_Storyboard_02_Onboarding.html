Storyboard 02 —
Admin onboarding & first run

Day one. An empty workspace has just been provisioned for Construgypsum and nobody has connected anything yet. This is the route from a blank room to the first agent run a human approves — the entry points, the empty states, and what happens when a connection refuses to cooperate.

Storyboard 01 showed a workspace already humming — connectors green, runs queued, reps in the field. That is the steady state, and it is the easy one to draw. This board draws the hard part: the beginning. Empty states are where most enterprise software loses the customer, and they are the screens nobody designs. Every frame here answers one question — what does the admin see before there is any data, and what is the single next thing we want them to do?

Frame note

The blank room is a designed screen, not an accident. It admits the emptiness, explains why, and offers exactly one next action.

  • Nav items that need data are dimmed, not hidden — the admin can see where they're headed.
  • The checklist is the storyboard's own route, shown to the user as progress.
  • Environment pill reads Setup, not Production — state is always visible.
  • One primary CTA per view, per the brand's CTA rule.

Frame note

The difference between a good and a bad error screen is how far it narrows the problem. Here the app shows exactly which of four things broke.

  • Fields that passed get a check; only the failing two go critical.
  • The diagnostic trace gives the technical person something to act on without a support ticket.
  • Offers the realistic next move — email the Odoo partner, since that is who holds the key.
  • "Nothing was saved" removes the fear of having half-broken the setup.

Frame note

This screen is the contract. It is what lets us say "the agent proposes, a human approves" and point at something concrete.

  • Writes off by default is the safe default and the honest sales story.
  • Scopes are written in business language, not API endpoints.
  • GL write is visibly present but off — scope guard made visual, matching the proposal.
  • Two layers of safety: the scope here, and the per-run approval gate later.

Roles are not decoration — they decide who can approve what, which is the whole safety model.

  • Panchito is an Operator: he approves inventory runs, and sees nothing else. That is Storyboard 03.

Frame note

Setup ends with something real — not a "you're all set" screen. The first thing the admin sees the agent do is a read-only run they approve.

  • "Nothing was written" is the sentence that earns trust on day one.
  • The nav has quietly switched from Get started to Dashboard — onboarding dissolves into the product.
  • This run hands off directly to Storyboard 03, where Panchito receives that digest.

What this board settles

Storyboard 01 assumed a working system. This one shows the twenty minutes that decide whether the client ever gets there. Three things fall out of it that the build needs: writes are off until someone turns them on, roles decide who can approve, and the field reps never log in — they meet the agent through WhatsApp, which is exactly where Storyboard 03 picks up.
